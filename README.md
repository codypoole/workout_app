# Handoff: Workout PWA — Plan, Track & Progress

## Overview
A phone-first workout tracker designed to be installed as a Progressive Web App. The product lets a user:
- Import a structured training plan as JSON (generated by Claude) that auto-expands into weeks → days → exercises → sets.
- Run a workout live, either one exercise at a time ("Focus") or as a full-day checklist ("Full day").
- Log weight × reps (or timed holds) per set, with an auto-starting rest timer between sets.
- Maintain a master library of exercises — browse, search, create, and edit on the fly.
- Swap, add, or delete exercises within a day.
- Track progress over time per exercise (estimated 1RM, total volume, PRs) with charts.

This document is self-sufficient: a developer who was not part of the design conversation should be able to implement the app from this README alone.

## About the Design Files
The files in this bundle are **design references created in HTML/React (via in-browser Babel)** — prototypes that demonstrate the intended look, layout, and behavior. **They are not a production codebase to ship as-is.**

The task is to **recreate these designs in a real, production environment** and **build out the backend** described below. There is no existing frontend codebase yet, so you should choose an appropriate stack. Recommended:
- **Frontend:** React (Vite) or Next.js with TypeScript, as an installable PWA (web app manifest + service worker). The prototype is already component-structured in a way that maps cleanly to a React app.
- **Styling:** The prototype uses hand-written CSS with CSS custom properties (design tokens). You can keep that approach or port the tokens into Tailwind / CSS Modules / styled-components — preserve the token values listed below either way.
- **Charts:** The prototype hand-rolls SVG line/bar charts. In production you may keep the lightweight SVG approach or substitute a chart library (e.g. Recharts / visx) — match the visual style described.
- **State/Data:** The prototype persists a single JSON blob to `localStorage`. Production should replace this with a real backend (see "Backend Requirements").

The HTML files double as a precise visual + behavioral spec. When in doubt about a measurement, color, or interaction, open the prototype and inspect it.

## App Structure & Navigation

The app is a single phone-sized surface (design canvas **393 × 852**, iPhone-class) with a **bottom tab bar** of four tabs:

| Tab | Purpose |
|-----|---------|
| **Today** | The active workout runner — Focus + Full-day modes, set logging, rest timer, finish workout. |
| **Plan** | The active plan: weeks → days overview, and the JSON import flow. |
| **Library** | Master exercise list — browse, search, filter, create, edit, delete. |
| **Progress** | Analytics — weekly volume, PR board, per-exercise strength trends + history. |

Two navigation variants exist as a tweakable option: a standard labeled **tab bar** (default) and a floating **dock** (icon-only, rounded, elevated). Ship the labeled tab bar as default.

Overlays are presented as **bottom sheets** (slide up from the bottom, rounded top corners, drag-grip, backdrop dimming + blur): Import plan, Swap/Add exercise, Exercise detail, and the New/Edit exercise editor.

## Screens / Views

### 1. Today — Workout Runner

**Header (all modes):**
- Eyebrow line: `<Week name> · Today` (or `· Recovery` on a rest day), mono, 11px, uppercase, letter-spacing 0.14em, color `--text-faint`.
- Day name as H1 (Archivo 800, 30px, letter-spacing −0.02em).
- Focus subtitle below (e.g. "Chest · Shoulders · Triceps"), 13px `--text-dim`.
- A **progress ring** top-right (58px, 6px stroke) showing completed sets / total sets, with `done` count over `/total` in the center.
- A segmented **pill toggle** beneath: **Focus** | **Full day** (hidden on rest days).

**Focus mode** (one exercise at a time):
- Nav row: back chevron (disabled on first), a row of progress dots (active dot is a 20px-wide lime pill; completed are 45% lime; pending are `--surface-3`), forward chevron (disabled on last).
- `Exercise N / Total` eyebrow + a "Complete" chip when all its sets are logged.
- Group color dot + `<Group> · <Equipment>` label.
- Exercise name as H1.
- Two ghost buttons: **Swap** (opens swap sheet) and **History** (opens exercise detail).
- A stacked list of **set rows** (see Set Row component).
- When the current exercise is fully logged and it isn't the last, a primary **Next exercise** button appears.
- **No Finish-workout button here** — finishing is intentionally only in Full-day mode.

**Full day mode** (zoom-out checklist):
- A card per exercise. Each card: a 34px index/checkmark badge, group dot + exercise name (clamped to one line), `X/Y sets · Equipment` sub-label, and per-set chips (showing `weight×reps` or timed `m:ss`; logged sets render in lime). Completed cards get a lime-tinted border.
- Each card has two trailing icon buttons: **swap** (swap icon) and **delete** (trash icon, danger color).
- Tapping the card body (name or chips) jumps into Focus mode at that exercise.
- Below the list: a dashed-outline **+ Add exercise** button (opens the Add sheet).
- Below that: the **Finish workout** footer (see Completion).

**Completion (Full-day mode only):**
- If not all sets are logged, the finish control is a **ghost** button labeled `Finish workout · {done}/{total} sets`, with a helper line "You can finish early — only logged sets are saved." If all logged, it's the **primary** (lime) button labeled "Finish workout."
- After finishing, the footer becomes a **Workout complete** card: trophy icon, "Workout complete" heading, `{done}/{total} sets · {volume} {unit} volume`, and two buttons — **Reopen** (undo completion; also rolls back the history row it wrote) and **Next workout →** (advance to the next non-rest day).

**Rest timer** (auto):
- When a set is logged with a `restSec`, a floating timer card appears near the bottom (above the tab bar): "REST" eyebrow + lime countdown `m:ss`, a progress bar that drains over the duration, and three controls — **+15s**, **pause/play**, **Skip rest**. Auto-dismisses at 0.

**Rest day:** centered empty state — circular flame icon, "Recovery day" heading, and a short supportive line. No runner, no finish button.

**Empty day** (all exercises deleted): centered "No exercises in this day yet." + a primary **Add exercise** button.

### 2. Set Row (core logging component)

Two visual states for weight/rep exercises, plus a timed variant.

**Collapsed (default):**
- Card-2 surface. Row: set index (mono, faint), then `{weight} {unit} × {reps}` in mono 18px/650, and (when not done) a faint "✎ edit" affordance pushed to the right.
- Trailing **check** icon-button: outlined lime when pending; solid lime (filled) when done. Tapping the check on a pending row logs it as-is; tapping a done row's check **un-logs** it.
- Done rows get a subtle lime-tinted background + border.

**Expanded editor** (tap the numbers / edit affordance):
- Card-2 with lime border. Header: "Set N" eyebrow + a small **Cancel** button (reverts and collapses).
- Two rows: **Weight** label + Stepper, **Reps** label + Stepper.
- A row showing live **Est. 1RM** `~{value} {unit}` (Epley formula) in lime mono.
- A full-width primary **Log set** button (sets the row done, collapses, triggers the rest timer).

**Stepper** (used here and in editors):
- Pill container (surface-2, 1px border, radius 12, height 48). Left **−** icon button, a **type-in numeric input** in the middle (mono 17px/600; `inputmode="decimal"`, selects-all on focus, commits on blur / Enter; clamps to `min`, default 0), optional unit suffix, right **+** icon button. Step is 5 for weight, 1 for reps.

**Timed variant:**
- Card-2 row: set index, the hold duration `m:ss` as a title, "Hold · {restSec}s rest" sub-label. A **Start** button begins a live countdown (button becomes **Stop**); reaching 0 marks the set done with a "Done" chip.

### 3. Plan

- Header: "Active plan" eyebrow, plan **name** as H1, and an **upload** icon button (lime-outlined) that opens the Import sheet.
- Chips: the plan **goal**, and "`{built}` of `{total}` weeks built".
- A horizontal scroll of **week chips** (active week is lime-filled).
- For the selected week, a card per **day**: `D{n}` index, day name (+ completed checkmark, + "Active" chip if it's the active day), `{count} exercises · {focus}` (or "Recovery day"), and a **Start** button on non-active, non-rest days that sets it active and jumps to Today. Each non-rest card lists its exercises inline (group dot + name, comma-separated, truncated).

**Import flow (bottom sheet, two steps via pill toggle):**

*Step 1 — Generate:*
- An info card explaining the prompt.
- A scrollable, monospace **preview of the generated Claude prompt**.
- A primary **Copy Claude prompt** button (writes to clipboard; shows "Copied" with a check for ~1.8s; has an `execCommand` fallback).
- A ghost button to jump to step 2.
- **The prompt is generated from the live library** — it embeds the full JSON schema, all rules (the `durationSec` convention for timed moves, `restSec`, rest-day shape, bodyweight = 0, progressive overload), an auto-generated list of every valid `exerciseId` with name/group/type, and a fill-in "MY PLAN REQUIREMENTS" section. See `seed.js → buildClaudePrompt()` for the exact text.

*Step 2 — Paste JSON:*
- A monospace textarea. As the user types, the JSON is **validated live**: JSON.parse errors, missing `name`/`weeks`, malformed `days`/`exercises` all surface as a red error card. **Unknown `exerciseId`s** surface as an amber warning (they still import, but show their raw id until added to the library).
- On valid input, a success card shows the plan name + counts (weeks / days / exercises / sets).
- Buttons: **Use sample plan** (loads the bundled sample), **Clear**, and a primary **Import & activate plan** (disabled until valid). Importing resets logs/completion and switches to Today.

### 4. Library

- Header: "Master library" eyebrow, "Exercises" H1, and a lime **+** icon button (new exercise).
- A **search** field (matches name or muscles) with a leading search icon.
- A horizontal scroll of **group filter chips**: All, Chest, Back, Legs, Shoulders, Arms, Core (active is lime).
- Results are **grouped by muscle group**, each section headed by a group dot + group name + count. Each exercise is a card: name (one line, clamped), a "Custom" chip if user-created, and `{equipment} · {muscles}` sub-label. **Exercise editor (sheet):** Name field; **Muscle group** chip picker; **Equipment** chip picker (Barbell, Dumbbell, Machine, Cable, Bodyweight, Kettlebell, Other); **Tracking type** as three selectable cards (Weight × reps / Bodyweight reps / Timed hold); a comma-separated **Muscles worked** field; a **Notes / cues** textarea. Primary **Save exercise**; on existing exercises, a danger **Delete exercise**.

**Swap / Add sheet** (shared component, `mode="add"` vs swap):
- Search field + a list of library exercises as cards (group dot, name, `{group} · {equipment}`).
- **Swap:** excludes the current exercise, sorts same-muscle-group first, tags those with a "Similar" chip; keeps the existing sets/reps. **Add:** includes everything, shows a + affordance, adds the exercise to today's day with 3 default sets (weight: 8-rep placeholders, or 30s holds for timed).

## Interactions & Behavior

- **Set logging:** tapping a pending check logs the set as planned; the expanded editor's "Log set" logs typed values. Either way the progress ring advances and, if the set has `restSec`, the rest timer auto-starts. Tapping a done check un-logs.
- **Rest timer:** 1s tick countdown; +15s extends; pause/play; Skip dismisses; auto-dismiss at 0. Reduced-motion / background behavior is acceptable to simplify.
- **Finish workout (Full-day only):** writes a history row per weight-type exercise that has ≥1 logged set (date + the logged sets), and marks the day completed. **Reopen** removes that day's history rows for today's date and clears completion. **Next workout** advances to the next non-rest day (wrapping across weeks).
- **Swap:** replaces `exerciseId` on the day's exercise, preserving its sets. **Add:** appends an exercise with default sets. **Delete:** removes the exercise and **re-indexes the day's set logs** around the removed index so existing logs stay attached to the right exercises.
- **Import:** builds a fresh plan object (assigning ids), resets logs + completion, sets the first non-rest day active, navigates to Today.
- **Live validation** on the paste field as described above.
- **Transitions:** sheets slide up (~0.26s cubic-bezier(.2,.8,.2,1)) with a fading backdrop; screens do a subtle pop/scale-in (~0.22s); the progress ring animates its stroke offset (~0.5s).

## State Management

The prototype keeps one in-memory state object, persisted to `localStorage` on every change. In production, model these as backend resources (see Backend Requirements) with optimistic client updates. Shape:

```
state = {
  library:    Exercise[],
  plan:       Plan,                       // the single active plan
  history:    { [exerciseId]: Session[] } // Session = { date: 'YYYY-MM-DD', sets: {weight,reps}[] }
  logs:       { [dayId]: { [exIndex]: { [setIndex]: { weight, reps, done } | { durationSec, done } } } },
  completedDays: { [dayId]: 'YYYY-MM-DD' },
  settings:   { theme, type, nav, unit }, // see tokens / tweaks
}
```

UI-local state (not persisted): current tab, active `{week, day}` pointer, focus index, current mode (focus/day), open sheet, rest-timer state.

**Calc helpers** (`seed.js`):
- `epley1RM(weight, reps)` = `reps===1 ? weight : round(weight * (1 + reps/30))`.
- `setVolume = weight*reps`; `sessionVolume = Σ setVolume`.
- `best1RMOfSession` = max Epley across sets; `bestSetOfSession` = the set with the highest Epley.

## Data Model / JSON Schema (the importable plan)

```jsonc
{
  "name": "Upper / Lower Hypertrophy",   // required
  "goal": "Hypertrophy",                  // optional
  "weeksCount": 6,                         // optional, defaults to weeks.length
  "weeks": [                               // required, non-empty
    {
      "name": "Week 1",
      "days": [
        {
          "name": "Push",
          "focus": "Chest · Shoulders · Triceps",   // optional
          "rest": false,                              // true => recovery day, exercises: []
          "exercises": [
            {
              "exerciseId": "bench-press",            // MUST match a library id
              "sets": [
                { "weight": 135, "reps": 8, "restSec": 120 },   // weight/bodyweight set
                { "weight": 155, "reps": 6, "restSec": 150 }
              ]
            },
            {
              "exerciseId": "plank",                   // timed exercise
              "sets": [ { "durationSec": 60, "restSec": 45 } ]  // durationSec, NOT weight/reps
            }
          ]
        },
        { "name": "Rest", "rest": true, "exercises": [] }
      ]
    }
  ]
}
```

Rules: `exerciseId` must exist in the library; timed exercises use `durationSec`; bodyweight uses `weight: 0`; `restSec` is rest *after* that set. The canonical prompt that instructs Claude to produce this (and embeds the live exercise-id list) is generated by `buildClaudePrompt()` in `seed.js`.

**Exercise (library item):**
```
{ id, name, group, equipment, type: 'weight'|'bodyweight'|'timed', muscles: string[], notes, custom: boolean }
```

## Design Tokens

Defined as CSS custom properties in `styles.css` (`:root` = default "midnight" dark theme). Exact values:

**Colors — dark (default):**
- `--bg` `#0a0a0b` (body also has a radial top-glow to `#18181c`)
- `--surface` `#141416`, `--surface-2` `#1c1c20`, `--surface-3` `#26262b`
- `--line` `rgba(255,255,255,0.08)`, `--line-2` `rgba(255,255,255,0.14)`
- `--text` `#f4f4f5`, `--text-dim` `#a1a1aa`, `--text-faint` `#6b6b73`
- `--accent` `#c8f135` (lime), `--accent-ink` `#11160a` (text on accent), `--accent-soft` = accent @ 16%
- `--danger` `#ff5a52`

**Accent theme variants** (swap `--accent` / `--accent-ink`): ember `#ff6a3d`, ice `#5aa2ff`, violet `#b69bff`. A light "paper" theme overrides the full neutral ramp (bg `#f3f1ec`, surface `#ffffff`, text `#19181a`, accent near-black `#1c1b1f`).

**Group colors** (muscle dots), via `oklch(0.72 0.16 <hue>)`: Chest 8, Back 200, Legs 140, Shoulders 48, Arms 280, Core 330.

**Radii:** `--radius` 18px, `--radius-sm` 12px; buttons 14px (lg 16px), icon buttons 12px, chips 999px.

**Spacing:** gap utilities at 4/6/8/10/12/16/20px; screen horizontal padding 16px; scroll regions get 96px bottom padding to clear the tab bar.

**Typography:**
- `--font-display` Archivo (500–900) — headings.
- `--font-body` Hanken Grotesk (400–700) — body/UI.
- `--font-mono` JetBrains Mono — all numbers (tabular), eyebrows, code.
- Type-theme variants: "technical" → Space Grotesk for display+body; "grotesk" → Bricolage Grotesque display + Hanken body.
- Scale: H1 30/800/−0.02em/1.1; H2 21/700; title 16/700; eyebrow mono 11 uppercase 0.14em; label 13 `--text-dim`; set numbers mono 18/650; big numbers (`.numbig`) Archivo 800 tabular.

**Shadows:** phone bezel uses layered rings; sheets/floating timer use `0 16px 40px -10px/-12px rgba(0,0,0,0.7)`.

**Buttons:** height 48 (lg 56); primary = accent bg + accent-ink text; ghost = transparent + `--line-2` border; danger = transparent + danger border/text; `:active` scales to 0.97.

## Backend Requirements (new build)
The prototype is client-only with seeded demo data. To productionize:

1. **Auth & user accounts** — each user owns their library, plans, logs, history, settings.
2. **Resources / endpoints** (suggested REST shape):
   - `Exercises` (library): list / create / update / delete. Seed the 20 defaults (in `seed.js`) per new user or as a shared global catalog + user-custom overrides.
   - `Plans`: create (incl. the JSON-import path — accept the documented schema, validate server-side, resolve `exerciseId`s), get active, list, update, delete, set-active.
   - `WorkoutLogs`: per (planDay, set) logged values + done flag; an endpoint to finish a day (which materializes history rows) and to reopen.
   - `History` / `Sessions`: per-exercise session records `{date, sets:[{weight,reps}]}` — written on finish; queried for the Progress charts. Consider computing 1RM/volume server-side or client-side (formulas above).
   - `Settings`: theme/type/nav/unit per user.
3. **Validation:** server must enforce the plan schema (mirror the client validator) and reject/flag unknown `exerciseId`s.
4. **PWA:** add a web app manifest (name, icons, theme-color `#0a0a0b`, display standalone, portrait) and a service worker for offline shell + cached assets. Offline-first logging with sync is a strong fit for a gym app — queue mutations locally and reconcile.
5. **Migration of calc logic:** port `epley1RM`, `sessionVolume`, `best1RMOfSession`, `bestSetOfSession` (in `seed.js`) verbatim so client and server agree.
